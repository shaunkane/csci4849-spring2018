      Slide 11

      Models of disability

      • Medical model: people with disabilities (PWD) are patients; our goal is to restore or replace function
      • Social model: What counts as "disability" is defined socially; people are naturally different, but society sets the bounds for what is considered normal and what is abnormal. Maybe we don't need to fix people.

      This is a pretty big idea. Worth thinking about. It's somewhat straightforward to imagine a perspective where there's nothing "wrong" with a disabled person. It's a whole different thing to take the perspective that the very idea of disability is a social construct.

      Slide 13

      Definitions change over time

      • Being nearsighted was not a significant disability until literacy become common
      • Changing norms around texting may benefit some (e.g., deaf people) and impair others (e.g., people with dyslexia)

      In the future, one could imagine stereo vision or VR sickness to count as disability.

      Slide 14

      WHO model of disability (2012)

      • Body functions and body structures describes how a person's body functions
      • Impairments refer to functional limitations of a person's body
      • Activity limitations refer to difficulties performing specific activities
      • Participation restrictions refer to difficulties participating in activities of daily life
      • (and all of this is affected by environmental factors)

      Link: WHO document

      The WHO has moved to more complex classifications; but this is a clear model that is useful for getting the big idea.

      For example, a deaf person might not be able to hear. We might consider that an impairment. But does it affect activities, like conversation? And if so, does that affect participation in school, work, social life, etc.?

        Slide 17

        Perspectives and solutions "What should we do about it?"

        • Rehabilitation technology - enable people to restore natural function (usually temporary)
        • Assistive technology - provide technology to solve an accessibility problem (usually permanent)
        • Special education - focus on access to education and empowerment
        • Legal perspective - focus on rights and societal supports
        • Whose responsibility is this problem?

        Ladner groups these in as models of disability. I think that's fine, but it is useful to separate out the underlying perspective on what disability is, from the perspectives of how to intervene

        Slide 18

        Design approaches

        • Assistive technology - developing technology to "repair" a disability or address an activity limitation
        • Universal design - build it from the ground up to accommodate the widest user group
        • Inclusive design - often more pragmatic than universal design (include more, but not everyone), emphasizes participation by excluded users in the design process

        Assistive technology might be a ramp attached to a building, while universal design would involve building the floor at the ground level so there are no steps to get in anyway.

        Slide 20

        More design approaches

        • Ability-based design - develop systems that model and adapt to a user's ability
        • Design for empowerment - emphasize empowering and increasing representation from marginalized groups
        • Value Sensitive Design - consider other values in design other than usability (e.g., independence, self-determination)
        • Example: if we could, should we replace ASL interpreters with ASL-translating smart gloves? Why or why not?

          Slide 28

          Ways to think about accessibility

          • Following standards - this helps our design work on a variety of current and future devices
          • Presenting content in multiple ways - since people may consume media in different ways, have we included the appropriate content for doing so?
          • Supporting various input methods - including mouse, touch, keyboard
          Slide 29

          More ways to think about accessibility

          • Using appropriate accessibility features of HTML (e.g., alt tags for images, ARIA labels for dynamic user interfaces)
          • Semantic UI code - should represent the logical structure of the interface, not just visual
          • Support end-user customization - e.g. font size, color contrast
          Slide 30

          Where we go wrong

          • Writing buggy UI code
          • Neglecting to include alternative representations (image descriptions, video captions) - or including lousy versions of these
          • Getting the visual presentation right but messing up the semantic representation (no semantic UI)

          Semantic = related to meaning in language

          • My favorite animals

            • Goat

          These two code snippets will render as mostly identical (close enough, anyway) on most browsers. But the structure of the code is very different.

          The left example, semantically, reads: a heading, then a list of items. The right one reads as 5 paragraphs. The left code tells us that the 4 animals are part of a set. The right code does not.

          A note on typography: For the bullet I used the Unicode bullet character (¥); you could also use the html code "¥". However, you pretty much never want to do this. Instead, specify your list using a list tag.

                Slide 45

                Text vs. nonspeech/tactile

                • Often we provide access through text, either spoken or Braille
                • Alternatives include tactile representations (such as tactile graphics) and sonification (representing data as sound)
                • We'll focus on text for now, visit other approaches later
                Slide 46

                Computer access

                • Typically use screen reader software
                • Provides description of operating system and applications as output
                • May map to audio, Braille, or both
                • Supports keyboard navigation as input
                • Must provide a mental model for navigating applications, responding to alerts, etc.

                Screen readers became much more complex once the Graphical User Interface became the primary way of using our computers.

                Slide 47

                Braille vs. audio output

                • With training, screen reader audio output can be up to 300+ WPM
                • Braille reading speed can be 200-400 WPM
                • Using Braille is silent, preserves aural attention
                • Braille literacy is low; this could create problems in the future
                • We should not assume that a blind person has access to a Braille display, or even that they read Braille

                Preferences may depend on the material or task.

                    Best practices for Braille

                    • Helpful to have Braille version available
                    • Don't assume that everyone has access to Braille devices, or reads Braille
                    • Presenting content as well-structured text enables the end user to decide on the format
                    Slide 54

                    Screen readers

                    • Most common form of computer access
                    • Represents applications and their components in a hierarchical form
                    • "an infinite set of nested lists"
                    • Typically paired with keyboard input (and not mouse input)
                    Slide 55

                    Screen reader use

                    • At top level: navigate between system and applications
                    • Navigate between different parts of application, based on underlying UI code
                    • Often jump around between parts of a document: headings, links, etc.
                    • May be combined with screen magnification, for low vision users)

                    You'll try one yourself!

                      Slide 57

                      How to think about screen reader use

                      • Text is linear and takes time to hear
                      • Little success in simultaneous speech
                      • Users may adjust verbosity in some applications
                      • Providing landmarks to navigate content is key
                      • Don't count on mouse input

                      What pops out in a screen reader is usually its position in the structure, rather than its visual characteristics.
